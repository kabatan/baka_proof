# Guardian Base Spec — GeometryFull2D v0.6 Execution-Locked Real Solver-Causal Full Pipeline

## 0. Purpose and acceptance meaning

This Base Spec exists because earlier GeometryFull2D implementations repeatedly satisfied textual or report-shaped release gates while avoiding the intended mathematical pipeline. The failure class is not one specific historical file. It is the general class of **Agent shortcut implementations**: implementations that construct artifacts, reports, rules, corpora, baselines, or causality summaries that look solver-causal but do not make solver output mathematically necessary for the final Lean proof.

The only permitted final claim is:

```text
V0.6_GEOMETRY_FULL2D_EXECUTION_LOCKED_REAL_SOLVER_CAUSAL_FULL_PIPELINE_READY
```

This claim means all counted positive `GeometryFull2DTarget:1.0.0` tasks are processed by this exact executable causal chain:

```text
sorry-only Lean theorem source
  -> Lean-side structured extraction for that exact theorem
  -> GeometryFull2DClaimSpec
  -> provider.solve in an isolated provider stage
  -> EngineOutputFull2D with nontrivial solver facts/constructions/certificates
  -> independent checker verifies each selected solver artifact
  -> SelectedSolverDerivationV3 assembled only from checked artifacts
  -> DerivationTargetMatcher proves the selected derivation entails the target hash
  -> compiler consumes SelectedSolverDerivationV3 and RuleRegistry contracts
  -> LeanPatchCandidateFull2D
  -> ProofWorker applies patch only inside MARP proof region
  -> FinalVerifyGate compiles generated candidate from scratch
  -> SolverBackedProofCertificateFull2D binds all upstream artifacts
  -> SolverCausalityLiveRunV1 performs live destructive reruns in fresh temp dirs
  -> all-baseline matrix and artifact-derived metrics
```

A report field is not evidence. A string such as `failed_as_expected`, `solver_causal_necessity=true`, `mutation_sensitive=true`, `replay_status=passed`, or `real_integration_flag=true` is valid only when linked to replayable command logs, temp directories, content-addressed inputs/outputs, and checker transcripts generated by the final release command.

## 1. Execution-locked release principle

v0.6 is **execution-locked**. A release checker must not pass because a static report says the pipeline would fail under mutation; it must execute the mutation and verify the result. A matrix must not pass because a summary says all baselines were evaluated; it must materialize actual records for every counted task and every required baseline. A solver artifact must not pass because it has fields named `fact`, `certificate`, or `checker`; it must be independently replay-checked.

The final closure command is the source of truth:

```bash
python scripts/check_release_acceptance_v0_6.py   --config configs/benchmark_runs/geometry_full2d_v0_6.yaml   --fresh-run   --fail-on-stale   --no-skip   --all-baselines   --live-mutations   --output docs/ai/changes/geometry-full2d-v0_6/evidence/release_acceptance_report.json
```

`--fresh-run` is mandatory for closure. A non-fresh replay run may be used only for debugging and must set `closure_allowed=false`. The final command must create a new run directory and fail if the target release run directory already contains nonempty records unless the command is explicitly in debug replay mode.

## 2. Generalized red-case failure class

A counted release success must fail if produced by any member of this class. The red-case suite must test the class by behavior, not only by filename or string pattern.

### RC-001 Target-fact provider

Provider or engine reads the final target and emits that target as an already-proved fact without a prior derivation.

Invalid selected artifact:

```yaml
fact:
  conclusion: <final target>
  premises: []
  construction_refs: []
  certificate_refs: []
  checker_transcript_ref: null
```

Valid only if the final target is the last step of a checked derivation whose earlier selected step contains a non-target intermediate fact, construction, case split, side condition, or certificate.

### RC-002 Naked target assertion

`SelectedSolverDerivationV3` contains the final target but no selected non-target intermediate, construction, certificate, case split, or side-condition discharge. This is invalid even if wrapped in a rule contract or checked by a schema checker.

### RC-003 Identity/direct-rule registry

A counted rule may not transform a fact into itself, may not have output equal to input modulo variable renaming, and may not be a facade theorem whose Lean template proves the final target in one step. Identity, reflexivity, and pure symmetry rules may exist only for B1/direct baseline or uncounted smoke tests, not as B2 solver-causal counted reasoning.

### RC-004 Proof-from-shape compiler

Compiler chooses proof text or rule plan from target expression, theorem family, task id, target_shape_id, grammar family, corpus category, source_ref, template_id, difficulty tier, or benchmark label.

Forbidden release decision patterns include:

```text
target_expr.startswith(...)
match target_expr
lookup_by_target_shape
proof_from_shape
proof_from_source
if theorem_family == ...
if baseline == ... and family == ...
```

### RC-005 Rule-list artifact synthesis

Engine output is generated from `used_rules`, compiler decisions, proof text, rule registry output, or any downstream proof-generation object. Provider/engine must run before compiler and cannot import compiler/rule-template/proof-worker/final-verifier modules.

### RC-006 Report-only causality

Causality report writes `failed_as_expected` or booleans without executing compiler + proof worker + FinalVerifyGate with mutated solver artifacts.

### RC-007 Family-coded baseline

Baseline success/failure is decided directly by theorem family, category, grammar family, target type, or hard-coded baseline id. Baseline outcomes must arise from the same pipeline with a declared component disabled.

### RC-008 Projection benchmark

External source provides points/predicates, but the counted Lean goal is replaced by an easier projection such as reflexivity, symmetry, midpoint-to-collinearity, or a one-step facade lemma. Projection tasks may be red-case fixtures only.

### RC-009 Checker omission

A Base Spec or Acceptance requirement exists but no release checker enforces it.

### RC-010 Checker whitelist

Checker detects a forbidden shortcut and suppresses it because of filename, directory, file role, comment, fixture tag, or role. Negative fixtures must be stored under `tests/red_cases/` or `docs/.../red_cases/`; release entrypoint scanning must never whitelist a release file. If a release file contains forbidden pattern hits, the release fails even if that file is a checker.

### RC-011 Sealed challenge collusion

Holdout generator imports or reads provider, compiler, rule registry, proof worker, final verifier, matrix runner, release checker, selected implementation artifacts, or prior release run records, or emits expected proof/rule/engine labels.

### RC-012 Stale evidence replay

Release accepts run records or artifacts not bound to current git head, selected implementation hash, corpus hash, config hash, checker hash set, and fresh run directory hash.

### RC-013 Rule hack

Rule registry contains fake broad families, many aliases of the same logical rule, unchecked rules, or rules whose checkers do not verify premises, side conditions, and conclusion independently.

### RC-014 Target-as-certificate

Certificate payload is just the target statement, target hash, target expression, or schema-normalized target with a checker name. Certificate must be independently replayable from hypotheses / construction / algebraic data / trace.


### RC-015 Checker-generated success artifacts

Release checkers or helper checkers generate success artifacts instead of executing provider / compiler / proof worker / final verifier stages. Checkers may generate only reports and red-case fixtures, never counted success artifacts.

### RC-016 Static-only green release

Final release passes from static scans, pre-existing reports, or summary fields without live provider, compiler, final verification, all-baseline matrix, and destructive mutation execution in the fresh release run directory.

### RC-017 B2-only matrix

Matrix executes B2 records only, or materializes non-B2 baselines from summaries, labels, or cached B2 results.

### RC-018 Direct lemma wrapped as intermediate

A direct facade lemma is wrapped in `have h_solver_intermediate : target := by exact ...; exact h_solver_intermediate` and counted as solver-intermediate reasoning.

### RC-019 Provider imports compiler

Provider or engine imports compiler, rule registry templates, proof generation, proof worker, final verifier, matrix, release checker, or corpus generator modules.

Every red case above is a release blocker. Summary rules: No target-fact provider. No proof-from-shape compiler. Passing the red-case suite is a prerequisite for any provider/compiler/matrix implementation claim.

## 3. Non-negotiable decisions

### DR-012-001 Red-case-first

Codex must implement and run the red-case suite before implementing provider, compiler, rule registry, matrix, or corpus expansion. The first accepted progress report is not a green pipeline; it is a red-case report proving all shortcut classes fail. A red case counts only if it fails for the specified K requirement and the red-case checker has a positive control proving the checker accepts a minimal non-red fixture.

### DR-012-002 Provider boundary

Provider/engine code may import only:

```text
standard library
math_auto_research base schema/resource modules
plugins.geometry_full2d_v0_6.provider_schema
plugins.geometry_full2d_v0_6.engine_checkers
allowed external solver adapters
```

Provider/engine code must not import:

```text
compiler
rule_registry
Lean patch generation
proof_worker
final_verify
matrix
metrics
corpus generator
release checker
previous release modules
```

Provider output must be persisted before compiler starts. Release records must include stage timestamps proving this order.

Provider may receive the goal for directed search, but provider selected artifacts cannot be the naked final target. At least one selected artifact in every counted B2 success must be non-target or must be a checked certificate/construction/case split that is not equivalent to writing the final target directly.

### DR-012-003 Engine artifact schema and checking

Each selected solver artifact must be one of:

```text
SolverFactV2
SolverConstructionV2
SolverCertificateV2
SolverTraceStepV2
SolverCaseSplitV2
```

Each selected artifact must include `independent_checker_ref`, and the checker transcript must be generated by a module that imports neither provider nor compiler. The checker must verify premises, side conditions, and conclusion from artifact data and ClaimSpec/hypotheses only. A checker that only validates JSON shape is invalid for selected artifacts.

### DR-012-004 Compiler API lock

Compiler release API is exactly:

```python
compile_derivation(
  theorem_anchor: TheoremAnchorV1,
  selected_derivation: SelectedSolverDerivationV3,
  rule_registry_snapshot: RuleRegistrySnapshotV1,
  side_condition_reports: list[SideConditionReportV1],
) -> LeanPatchCandidateFull2D
```

`TheoremAnchorV1` contains theorem name, proof-region markers, statement hash, and variable binder map. It must not contain raw target expression, theorem family, task id, corpus category, difficulty tier, source_ref, target_shape_id, or benchmark labels. Final target matching is performed by `DerivationTargetMatcher`, which returns only a target-hash equivalence result and no proof tactic or target expression string.

### DR-012-005 No target-shape proof strategy

Compiler may not choose a proof strategy from final target shape. It may choose proof steps only by iterating over selected solver derivation steps and mapping each checked rule/certificate to Lean lemmas declared in the rule registry. If selected derivation is empty or only contains target fact, compiler must fail.

### DR-012-006 Rule registry semantics

Each counted rule contract must include:

```yaml
RuleContractV2:
  rule_id: ...
  family: ...
  input_patterns: [...]
  output_pattern: ...
  side_conditions: [...]
  lean_lemma: ...
  lean_lemma_type_hash: sha256:...
  independent_rule_checker: ...
  positive_fixtures: [...]
  negative_fixtures: [...]
  unsupported_variants: [...]
  counted_release_rule: true | false
```

A counted rule must not be identity, pure target assertion, or direct facade proof. The registry must include at least 35 counted rules across at least 15 families. Aliases do not count. A rule counts only if used in a final theorem and passes rule-level mutation tests.

### DR-012-007 Live destructive causality

For every B2 counted final theorem success, release acceptance must run `SolverCausalityLiveRunV1` in a fresh temp directory:

```text
positive_control
remove_selected_artifact
corrupt_non_target_intermediate
corrupt_construction_or_certificate
unsupported_rule_mutation
side_condition_mutation
remove_checker_transcript
```

Each mutation reruns compiler, proof worker, and final verifier. It must record command logs, input/output hashes, patch hash, final verify status, and whether the same final theorem was counted. If any mutation still produces the same counted final theorem, release fails.

### DR-012-008 All-baseline actual matrix

For each counted task, records must exist for B1, B2, B5, B6, B7, and B8 if enabled. `run_full2d_matrix_v0_6.py --execute-all` must materialize all records in the fresh run directory. It must not skip records in closure mode. It must not compute outcomes from labels. Disabled baselines must remove actual artifacts or stages, and failures must be recorded as `StageFailureReportV1` or `DisabledStageReportV1` from actual stage execution.

### DR-012-009 Corpus independence

Counted positives may be:

```text
ExternalGoalPreserved
SealedAdversarialHoldout
UserReviewedGoal
```

`SealedAdversarialHoldout` must be generated after provider/compiler/rule-registry freeze by a generator that imports no release implementation code and contains no expected proof labels, rule ids, engine roles, or target-shape proof menus.

The challenge generator must be declarative and randomized from a release seed generated by the final acceptance command after implementation freeze. The seed and generator transcript must be stored. The generator may define target relation families and grammar productions, but not expected proof tactics/rules/engine roles. It must emit sorry-only theorem statements and metadata needed for diversity accounting only.

### DR-012-010 Statement diversity

Release positive corpus must satisfy:

```text
counted positives >= 1200
negative/malformed/target-outside >= 200
unique normalized target ASTs >= 900
unique hypothesis-target dependency signatures >= 700
near duplicate fraction <= 0.05
direct/facade lemma eligible fraction <= 0.05
construction/case/certificate-required tasks >= 600
multi-step derivation required tasks >= 800
```

No family may exceed 20% of counted positives. No target predicate family may exceed 30%.

### DR-012-011 External source availability is not a HardBlocker

External sources are used when available. If unavailable, a checker must produce `ExternalSourceAvailabilityReportV2` with command transcripts / HTTP status / local path checks. Missing external sources reduce the ExternalGoalPreserved floor but must be replaced by sealed adversarial holdout tasks. This prevents environment-dependent nonessential blockers while preserving challenge strength.

### DR-012-012 Fresh evidence only

Closure requires a release run generated by the final release command after active spec approval. Freshness report must bind:

```text
git head
git status hash
selected implementation hash
provider/engine/compiler/rule-registry hashes
corpus manifest hash
challenge seed hash
config hash
checker hash set
fresh run directory hash
all artifact refs
```

### DR-012-013 Acceptance coverage lock

Every K requirement in Acceptance must map to an executed checker and at least one red-case test or dynamic invariant. `check_acceptance_coverage_v0_6.py` must fail if any K lacks enforcement. The release report must list `K -> checker -> result -> evidence_ref`.

### DR-012-014 Closure claim ceiling

`CLOSURE.md` must be generated by release acceptance only after all checks pass. Hand-written closure cannot grant the final claim. `check_closure_claim_ceiling_v0_6.py` must fail if closure claims more than release report supports.

## 5. Minimum release metrics

Final release requires:

```text
B2 overall final theorem rate >= 0.70
B2 solver-causal success fraction = 1.00
B2 live destructive mutation pass fraction = 1.00
B2 non-target intermediate success fraction >= 0.70
B2 construction/case/certificate success fraction >= 0.50
B2 direct/facade lemma fraction <= 0.05
B2 used counted rule families >= 15
B2 all release-critical engine roles used in at least one counted success
B2 - B1 overall advantage >= 0.20
B2 - B5 construction subset advantage >= 0.10
B2 - B6 algebraic/metric/certificate subset advantage >= 0.10
B2 - B7 order/case subset advantage >= 0.05
```

Thresholds are evaluated on artifact-derived actual records only.

## 6. HardBlocker / ReleaseBlocker policy

Codex stops only for HardBlockers:

```text
schema contradiction
unsafe proof-use path that could make false Lean theorem count
missing user permission for destructive repo operation
license restriction blocking use of required external data
```

Everything else is ReleaseBlocker or WorkDebt. Codex must record it and continue with next unblocked work package. ReleaseBlockers block final closure but not implementation progress.

## 7. Forbidden completion claims

Until all v0.6 release gates pass, forbidden claims include:

```text
full pipeline completed
real solver-causal pipeline ready
high-performance geometry prover
V0.6_GEOMETRY_FULL2D_EXECUTION_LOCKED_REAL_SOLVER_CAUSAL_FULL_PIPELINE_READY
```

Allowed interim claim:

```text
implementation is in progress under v0.6; release blockers remain open.
```
